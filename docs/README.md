## 🚀 快速开始

### 👥 不同角色的阅读建议

#### 🎨 设计师
1. 先阅读 [全局设计规范](design/全局设计规范.md) 了解设计系统
2. 查看 [组件设计规范](design/组件设计规范.md) 了解组件约束
3. 参考 [设计实现指南](design/设计实现指南.md) 了解技术实现

#### 💻 前端开发者
1. 从 [开发指南](development/开发指南.md) 开始搭建环境
2. 学习 [全局设计规范](design/全局设计规范.md) 和 [设计实现指南](design/设计实现指南.md)
3. 深入理解 [系统架构](architecture/系统架构.md) 和 [数据模型](architecture/数据模型.md)

#### 🏗️ 架构师/后端开发者
1. 重点关注 [系统架构](architecture/系统架构.md) 和 [数据模型](architecture/数据模型.md)
2. 了解 [业务指标计算](business/业务指标计算.md) 的业务逻辑
3. 查看 [API参考](reference/API参考.md) 了解接口设计

#### 📊 产品经理/业务专家
1. 从 [业务指标计算](business/业务指标计算.md) 了解核心业务逻辑
2. 学习 [权重分配规则](business/权重分配规则.md) 理解目标拆解
3. 查看 [组织模式](architecture/组织模式.md) 了解机构体系

#### 🔧 测试人员
1. 重点学习 [数据校验标准](business/数据校验标准.md) 的质量标准
2. 参考 [故障排除](reference/故障排除.md) 了解常见问题
3. 查看各业务层文档了解测试要点

## 📋 文档使用规范

### 🔄 文档更新流程
1. **设计变更**: 更新 `design/` 层文档 → 同步到实现指南
2. **架构变更**: 更新 `architecture/` 层文档 → 影响分析
3. **业务变更**: 更新 `business/` 层文档 → 计算引擎调整
4. **实施变更**: 更新 `development/` 层文档 → 任务跟踪

### 📖 阅读顺序建议
- **新人入门**: README → 开发指南 → 全局设计规范 → 系统架构
- **功能开发**: 业务指标计算 → 数据模型 → 组件设计规范 → API参考
- **问题排查**: 故障排除 → 相关业务文档 → 系统架构
- **设计评审**: 全局设计规范 → 组件设计规范 → 业务需求文档
- **文档维护**: 文档协作指引（Agent） → 文档更新流程 → 版本更新日志

### ⚠️ 重要注意事项
- 🎯 **全局设计规范** 是所有UI实现的唯一权威源
- 🏗️ **系统架构** 是技术决策的权威依据
- 📈 **业务指标计算** 是计算口径的权威定义
- 📋 所有文档必须保持与代码实现的一致性
